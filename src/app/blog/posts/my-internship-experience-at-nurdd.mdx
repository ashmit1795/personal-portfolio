## About Nurdd

Nurdd is an AI-powered marketing intelligence platform that helps brands make smarter creator marketing decisions.

The platform enables businesses to discover creators, analyze influencer performance, measure AI search visibility, and understand marketing insights through large-scale data processing and AI-driven workflows.

Behind every creator profile that users search lies an enormous amount of engineering.

Data has to be collected from multiple platforms, enriched with analytics, processed through different services, stored efficiently, and finally served with low latency.

During my internship, I had the opportunity to work on exactly these kinds of problems.

## Solving a 5-Minute Request Problem

One of the first large engineering challenges I worked on involved creator profile generation.

At first glance, it sounded simple.

A user searches for a creator.

The backend returns the profile.

But behind that request, the platform needed to:

* fetch creator information
* collect hundreds of social media posts
* retrieve engagement analytics
* calculate fake follower metrics
* aggregate insights from multiple external services
* process and normalize the data

If the creator already existed in our database, the response was almost instantaneous.

But if it was a completely new creator, all of these operations had to happen before returning a response.

That process could easily take five to six minutes.

Clearly, keeping a user waiting for that long wasn't an option.

---

## Designing an Asynchronous Architecture

Instead of blocking the request until all processing finished, I designed and implemented an asynchronous workflow using BullMQ and Redis.

The request flow became much simpler:

1. Check whether the creator already exists.
2. If yes, return immediately.
3. Otherwise, enqueue the request.
4. Background workers process the creator independently.
5. Notify waiting users once processing completes.

The frontend remained responsive while the heavy lifting happened in the background.

That single architectural decision dramatically improved the user experience.

---

## The Concurrency Problem Nobody Notices

One interesting challenge appeared after implementing the queue.

Imagine ten users searching for the same creator simultaneously.

Without additional safeguards, the system would happily create ten identical jobs.

That meant:

* ten expensive API calls
* duplicate processing
* higher infrastructure costs
* unnecessary load

This is one of those production problems that rarely appears in classroom assignments but happens all the time in real systems.

To solve it, I implemented a distributed locking mechanism using Redis.

Only one background job could exist for a creator at any given time.

Instead of creating duplicate jobs, subsequent requests were added to a Redis-backed waiter list.

When processing finished, every waiting request received the completed result.

The system became significantly more efficient while reducing duplicate work.

## Reliability Matters More Than Success

Another lesson I learned was that production systems must expect failure.

External APIs fail.

Networks fail.

Services timeout.

Servers restart.

Instead of assuming everything would succeed, I built the processing pipeline to recover automatically.

The workflow included:

* exponential retry backoff
* automatic retries
* safe job recovery
* dead letter queues for persistent failures

Rather than crashing silently, failed jobs could now be inspected, debugged, and replayed.

Building for failure became just as important as building for success.

## Scaling YouTube Creator Processing

Another major area I worked on involved YouTube creator ingestion.

Using the YouTube Data API v3, I helped build pipelines responsible for:

* fetching creator metadata
* collecting analytics
* enriching creator information
* handling large-scale creator onboarding

The architecture again relied on asynchronous processing rather than synchronous requests.

As the platform continued growing, these queue-based workflows helped keep ingestion scalable while reducing request latency.

## Stepping Into AI Systems

One of the most exciting opportunities during my internship came when I started working on Nurdd's AI chatbot, Nia.

Initially, the chatbot relied on multiple sequential LLM calls.

Each user message flowed through several separate stages:

* content filtering
* query rewriting
* context resolution
* intent classification
* routing

Although functional, every additional LLM call increased latency, token usage, orchestration complexity, and opportunities for hallucination.

I wasn't originally hired as an AI engineer.

In fact, most of my experience was with JavaScript rather than Python.

But that's one thing I appreciated about Nurdd.

Good ideas were welcomed regardless of who proposed them.

I spent time learning the required Python ecosystem, understanding the OpenAI SDK, and mapping concepts from the JavaScript SDKs I had already worked with.

Eventually, I redesigned the chatbot into a single-agent architecture using tool calling.

Instead of multiple sequential LLM invocations, one central model could dynamically invoke tools whenever it needed additional information.

These tools directly queried backend services and databases, giving the model access to fresh context without unnecessary inference steps.

The improvements were immediately noticeable:

* lower latency
* fewer model calls
* better context retention
* improved conversational accuracy
* simpler orchestration
* lower token consumption

It was one of the most rewarding engineering problems I worked on because it combined backend engineering with applied AI.

## Making Conversations More Natural

Another exciting contribution involved adding voice capabilities.

Users could now interact using voice instead of typing, making conversations significantly more natural and accessible.

Watching a system evolve from text-only interactions into multimodal conversations was incredibly satisfying.

## Ownership Is More Than Writing Code

Perhaps the biggest lesson from this internship wasn't technical.

It was learning what ownership actually means.

Ownership meant asking:

* What happens if this service fails?
* How will this behave under load?
* What happens if two users do the same thing simultaneously?
* Can this architecture scale six months from now?
* How do we debug this in production?

I wasn't simply implementing assigned tasks.

I was expected to think through the engineering decisions behind them.

That responsibility accelerated my growth far more than simply writing more code ever could.
